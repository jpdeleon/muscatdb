Pipeline Execution Details

1 FITS Header Scanning
Extracts metadata from raw scientific FITS files. It excludes calibration frames (dark/flat/bias) and writes clean logs.
muscat-db scan <inst> <yymmdd>
2 Database Assembly
Ingests the scanned CSV files, building indices and tables in a unified, optimized SQLite database.
muscat-db build-db
3 Interactive Browsing
Allows real-time target searches (with regex support), logs inspection, and observation schedules directly from the Web GUI.
muscat-db serve
4 Photometry Reduction
Launches a calibrated background reduction using the prose pipeline to align frames, build PSF models, and extract raw lightcurves.
5 Transit Fitting
Runs a PyMC MCMC sampler to model the transit curve, compute planetary/stellar parameters, and output high-quality fit plots.
6 Job Monitoring
Tracks active, pending, or completed photometry and transit fitting background processes. Shows logs and error tracebacks.
7 Ephemeris O-C Analysis
Combines completed transit-fit midpoint measurements across datasets, fits a linear ephemeris, and renders an interactive O-C plot for timing-variation inspection.

Frequently Asked Questions

Detailed explanations of pipeline internals, configuration options, and common workflows.

▼

Database

How does the database work? +

The database is a SQLite file (muscat.db) that stores observation metadata extracted from FITS headers. The pipeline has two stages:

  1. Scan — muscat-db scan <inst> <yymmdd> reads every FITS file in the raw data directory, extracts HEADER keywords (OBJECT, FILTER, RA, DEC, EXPTIME, etc.), and writes a CSV log per CCD under /ut3/muscat/obslog/. Calibration frames (bias, dark, flat) are excluded; only science frames are recorded.
  2. Build — muscat-db build-db ingests all scanned CSV logs into SQLite with indexes on instrument, date, object, and filter. The resulting DB powers the web GUI's browse, search, and target discovery features.

The DB also persists background job state (photometry and transit fit runs) so running jobs survive a server restart.

How often does the database get updated? +

The database is rebuilt automatically once per day at 5:00 PM local time via a cronjob:

30 17 * * * cd /raid_ut2/home/jerome/github/research/project/muscat-db && uv run muscat-db build-db >> /raid_ut2/home/jerome/github/research/project/muscat-db/logs/build-db.log 2>&1

This re-scans all CSV observation logs under /ut3/muscat/obslog/ and rebuilds the summaries and targets tables to reflect any new or corrected scans. Log output is written to logs/build-db.log for troubleshooting.

To trigger a manual rebuild at any time, run muscat-db build-db from the repo root.

Photometry

How does the photometry pipeline work? +

The prose pipeline processes calibrated FITS frames through a sequence of steps:

  1. Band grouping — science frames are sorted into photometric bands using the FILTER header keyword
  2. Reference frame creation — one frame (typically the middle frame or the first if --ref_band is used) serves as the reference for alignment
  3. Source detection — point sources are identified on the reference frame
  4. Target identification — the target is located via WCS matching against the resolved sky coordinates (from MAST/SIMBAD name resolution)
  5. Gaia-based aperture sizing — annulus and aperture radii are set to avoid contaminating sources
  6. Parallel reduction — each frame is aligned, centroided, and photometered in parallel
  7. Differential photometry — automatic selection and weighting of comparison stars (Broeg et al. 2005)
  8. Time correction — JD-UTC is converted to BJD-TDB
  9. Export — per-band CSV lightcurves, PNG diagnostics, optional GIF animation
How does the Broeg et al. 2005 autodiff algorithm work? +

The automatic differential photometry algorithm (implemented in prose) optimally selects and weights comparison stars to minimize the scatter in the target's differential lightcurve:

  1. Candidate pool — all detected stars (excluding the target) are considered as potential comparison stars
  2. Iterative selection — starting with a first guess of the target lightcurve, the algorithm iterates between:
    • Building an optimal reference lightcurve as a weighted average of comparison stars
    • Computing the target's differential lightcurve by dividing its raw flux by this reference
    • Updating the comparison star weights based on their correlation with the current reference
  3. Weights — stars with lower intrinsic scatter and higher correlation with the ensemble receive higher weights. Stars affected by systematic noise (e.g., airmass trends, seeing variations) are down-weighted automatically
  4. Convergence — the iteration continues until the target lightcurve stabilizes, typically within a few iterations

The key advantage is that no manual selection of comparison stars is needed — the algorithm handles both the choice and the relative weighting, and it can exclude bad comparison stars that would otherwise introduce noise.

How are comparison stars determined and how is it related to max_num_stars? +

The comparison stars used to compute the differential lightcurve of the target are determined as follows:

  • max_num_stars Relation — During source detection, the pipeline is configured to detect and track exactly up to max_num_stars (default 10) of the brightest point sources across all frames. All comparison star candidates are chosen from this subset of tracked stars.
  • Manual Selection — If explicit comparison star indices (--cID) are provided by the user, the pipeline validates that they are in-range and do not belong to the avoided/excluded list before using them directly.
  • Automatic Selection (Broeg et al. 2005) — If no explicit IDs are provided, the pipeline executes the autodiff() algorithm. This algorithm dynamically determines and weights comparison stars from the candidate pool (excluding the target and any avoided stars) to minimize the scatter in the target's differential lightcurve.
  • Avoided Stars — Stars can be manually excluded via --avoid_cids. Additionally, the pipeline automatically detects stars near the CCD edge (edge_cids) and adds them to the avoid pool to prevent edge-effect systematics from contaminating the comparison pool.
What is the difference between a test run and a full run? +
  • Test run — processes only the first 10 frames per band (configurable via --test_run_frames). The pipeline runs all stages (alignment, source detection, centroiding, photometry) but produces a short lightcurve. Used to verify settings, check alignment quality, and preview results before committing to a full reduction.
  • Full run — processes every science frame in the data directory. Required for publishable lightcurves and transit fitting.

You can always re-run with --overwrite to replace a test run with a full run later.

How do I interpret the diagnostic plots? +
  • Reference image (*_ref.png) — the stacked reference frame with detected sources circled and the target marked. Verifies that sources are correctly identified and the target is at the right position.
  • Apertures (*_apertures.png) — overplot of aperture radii and sky annuli on the reference. Check that the inner annulus excludes nearby stars and the outer annulus is within clean sky.
  • Alignment (*_alignment.png) — shows the shift (dx, dy) computed for each frame relative to the reference. Large or discontinuous shifts may indicate guiding problems or a bad reference frame.
  • Lightcurves (lightcurves.png) — multi-band raw and differential lightcurves side by side. The raw flux should be stable; the differential should be flat for non-variable targets. Trends in the differential suggest poor comparison star selection.
  • Covariates (covariates.png) — systematics summary: FWHM, airmass, background, and centroid positions across the sequence. Correlations between these and the lightcurve can guide detrending choices.
  • Stacks (stacks.png) — grouped thumbnail grid of one cutout per frame per band to quickly flag outliers (cosmic rays, clouds, tracking losses).
How does Gaia-based aperture sizing work and when should I override it? +

The pipeline queries the Gaia DR3 catalog around the target position to identify contaminating sources:

  • Annulus sizing — the inner radius (Rin) is set to 6 × FWHM, the outer (Rout) to 10 × FWHM (capped at 100 pix). These values are then adjusted inward if a Gaia source brighter than 10% of the target flux falls within the annulus.
  • Aperture grid — a series of radii from 1 × FWHM up to Rin in 2-pixel steps is generated, and the one that minimizes scatter on the brightest stars is selected individually per image.

When to override: for crowded fields where Gaia sources are too numerous (the annulus becomes unusably small), or for isolated bright targets where the heuristic is overly conservative. Use --aper_radii MIN,MAX,DR and --annulus RIN,ROUT to set custom values.

Why does photometry need a WCS solution? +

WCS (World Coordinate System) maps pixel coordinates to sky coordinates (RA, Dec). The pipeline needs it for two things:

  • Target identification — the resolved target coordinates (from MAST/SIMBAD name resolution) are projected onto the image via WCS to locate the target's pixel position among the detected sources. Without WCS, the pipeline cannot automatically identify which star is the target.
  • Gaia query — the target's sky position is used to query Gaia DR3 around the field for contamination analysis.

If WCS is missing from calibrated headers, --target_coord RA Dec can be passed instead — but source identification then relies on source ordering (brightest = target), which is unreliable for crowded fields.

How is barycentric correction (BJD-TDB) computed? +

The pipeline converts observation times from JD-UTC to BJD-TDB to remove the light-travel time variation caused by Earth's orbital motion around the Solar System barycenter:

  • Default method — astropy's light_travel_time function computes the barycentric correction using the target's sky coordinates and the observatory geocentric position (derived from the FITS header's SITE keyword or instrument defaults like Teide Observatory for MuSCAT2).
  • barycorrpy option — when --use_barycorrpy is enabled, the barycorrpy package (a Python wrapper for Jason Eastman's barycorr) is used instead. It accounts for additional effects including Shapiro delay and Earth's precession/nutation.

The difference between the two methods is typically <0.1 seconds for MuSCAT observing cadences. barycorrpy is recommended for sub-minute transit timing measurements.

How is the target_id (tID) auto-detected and when should I override it? +

The pipeline identifies the target star among all detected sources by matching the resolved target coordinates (from MAST) against the WCS-transformed positions of every detected source, selecting the nearest match within a tolerance.

When to override — if the WCS solution is poor, if the field contains multiple stars at similar coordinates, or if MAST resolution failed and the fallback assumed the brightest star is the target. Set --tID N where N is the 0-indexed source number from the reference image (circled labels on the *_ref.png plot).

How does the pipeline handle narrowband filters differently? +

The pipeline auto-detects narrow-band observations by inspecting the obslog CSVs for known narrow-band filter names (g_narrow, Na_D, i_narrow, z_narrow). When narrow bands are present:

  • The default band list is expanded from the four broadband filters (gp, rp, ip, zs) to include all detected narrow bands.
  • Narrow-band frames are reduced with the same pipeline steps (alignment, centroiding, photometry) as broadband frames.
  • Products are named with the narrow-band token so they don't collide with broadband outputs.

To override the auto-detection, specify bands explicitly with --bands gp rp Na_D.

How do I re-reduce a target with different settings? +

The --overwrite flag (enabled by default in the GUI) allows re-running the pipeline over existing results. This is useful when you want to change parameters (aperture radii, bands, reference frame) without manually cleaning the output directory. Simply launch again from the photometry page with the new settings — existing product files are replaced.

Note: previous outputs are lost. If you want to preserve an older reduction, rename the results directory or run on a different date/target combination.

What are the default photometry parameters and where are they configured? +

Default parameters are defined in src/muscat_db/photometry.py in the RUN_DEFAULTS dictionary. Key defaults:

  • bands: gp, rp, ip, zs
  • aper_radii: auto (Gaia heuristic)
  • annulus: auto (Gaia heuristic)
  • aper_unit: pix
  • max_num_stars: 10
  • min_star_separation: 10.0 pix
  • cutout_size: 35 pix
  • bin_size_minutes: 10.0
  • gif_stride: 100
  • test_run_frames: 10
  • make_gif: True
  • overwrite: True

These can be overridden individually in the GUI form or via command-line flags.

What is the difference between aper_unit pix and fwhm? +
  • pix — aperture radii and annulus dimensions are specified and applied in absolute pixel units. The same pixel values are used for every frame regardless of seeing conditions.
  • fwhm — radii are specified as multiples of the FWHM measured on each frame. This adapts the aperture size to the actual seeing: in good seeing (small FWHM), the aperture shrinks to exclude more sky; in poor seeing (large FWHM), it expands to capture more of the star's flux.

Use pix when the PSF is stable and you want a fixed physical aperture. Use fwhm when seeing varies significantly across the sequence — it produces more consistent photometry at the cost of slightly more complex interpretation.

How are saturated stars handled by the pipeline? +

The pipeline reads the saturation level from the FITS header (SATURATE keyword or BSCALE/DATAMAX). During source detection, pixels above a configurable fraction of the saturation level are flagged. However:

  • Detection — saturated sources are still detected but their centroids may be biased. The pipeline uses Gaussian2D centroiding which degrades on saturated cores.
  • Photometry — aperture photometry on saturated stars underestimates the true flux because flux is lost in the saturated core. The pipeline does not apply a non-linearity correction.
  • Comparison stars — the autodiff algorithm will naturally down-weight saturated comparison stars because they produce noisier differential lightcurves.

For heavily saturated targets, use shorter exposure sequences or apply a non-linearity correction externally before running the pipeline.

How does the ref_band alignment strategy affect photometric precision? +

The --ref_band flag controls whether all bands are aligned to a single reference band or each band self-references its own first frame:

  • Per-band self-reference (default, --ref_band not set) — each band creates its own reference frame from its first science frame. Alignment is within-camera only. Recommended for multi-camera instruments (MuSCAT3/4) where each band uses a separate CCD, because cross-camera registration would introduce interpolation artifacts from slightly different pointing and distortion.
  • Single reference (--ref_band gp for example) — all bands align to the reference band's middle frame. This can introduce systematic offsets between bands because different cameras have different pixel scales, rotations, and distortions. The resulting lightcurves may show band-dependent systematics.

For MuSCAT2 (single CCD with filter wheel), both strategies work because all bands share the same detector. Per-band self-reference is still the default and generally recommended.

Transit Fitting

How does the transit fitting pipeline work? +

The timer transit fitting pipeline models the transit lightcurve using PyMC's NUTS sampler:

  1. Configuration — YAML files (fit.yaml, sys.yaml) specify stellar and planetary parameters, priors, detrending covariates, and sampler settings.
  2. Data loading — photometry CSVs are read and merged with covariate data (FWHM, airmass, centroid offsets).
  3. Prior initialization — stellar parameters (Teff, logg, metallicity) are sourced from the TOI catalog or manually entered; planetary priors (period, T0, duration, impact parameter) are set from known ephemerides or the TOI catalog.
  4. Model assembly — a PyMC model is built with the transit model (batman or starry), limb darkening (quadratic or nonlinear), and systematics model (linear or GP).
  5. Pre-optimization — L-BFGS finds the maximum a posteriori (MAP) solution to initialize the sampler.
  6. MCMC sampling — NUTS sampler runs multiple chains (default 4) with several thousand tuning and production steps.
  7. Post-processing — residual outliers are sigma-clipped, information criteria (BIC, AIC) are computed, and diagnostic plots (corner, trace, LD profile, fit) are generated.
How does the parameter query from the TOI / NASA Exoplanet Archive work? +

The transit fit page has a Query Archive button that fetches stellar and planetary parameters from two sources:

  • TOI Catalog (source=toi) — queries the TESS Objects of Interest table via the Exoplanet Archive TAP service. The target name is cleaned by removing the TOI prefix, leading zeros, and .01 suffix (e.g. TOI07475.01 → 7475), then matched against the toi column with fallback queries on toidisplay and substring matching. Returns Teff, logg, orbital period, transit midpoint, and transit duration.
  • NASA Exoplanet Archive (source=nasa) — queries the pscomppars (Planetary Systems Composite Parameters) table by pl_name or hostname. Returns the same parameters plus metallicity, radius ratio, and impact parameter, with literature references.

The fetched values are used to pre-populate the priors in the YAML configuration, which can be edited before running the fit.
